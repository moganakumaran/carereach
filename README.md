## Prerequisites (important — read before `bundle deploy`)

This bundle deploys with the Databricks CLI **direct deployment engine** (no
Terraform). Set this env var for every `bundle` command (export it in your shell):

```bash
export DATABRICKS_BUNDLE_ENGINE=direct
```

Why direct (not the default Terraform engine): (1) the bundle manages Unity
Catalog resources, which require the direct engine; and (2) the CLI's bundled
Terraform download currently fails signature verification in some environments
(`openpgp: key expired`). The direct engine avoids both. (If you ever do use the
Terraform engine, install Terraform via `brew tap hashicorp/tap && brew install
hashicorp/tap/terraform` and set `DATABRICKS_TF_EXEC_PATH`/`DATABRICKS_TF_VERSION`.)

**Auth** — OAuth profile `mdp` against the hackathon workspace:

```bash
databricks auth login --host https://dbc-c5bb8304-1213.cloud.databricks.com -p mdp
```

Pass `-p mdp` to bundle commands (or auth to that host with any profile).

**One-time manual step — create the `mdp` catalog in the UI.** This metastore
uses UC **Default Storage**, and Default-Storage catalogs can only be created
through the Databricks UI (the API/CLI/bundle create path is rejected with
*"Please use the UI to create a catalog with Default Storage"*). In Catalog
Explorer: **Create catalog → name `mdp` → Standard (Default Storage)**. The
bundle then manages everything *under* `mdp` (schemas `bronze`/`silver`/`gold`/
`ops` and the `bronze.raw` volume) via `databricks bundle deploy`.

> dev and prod intentionally share the single stable `mdp` catalog, so the dev
> target omits `mode: development` (which would otherwise prefix schema names
> with `dev_<user>_`). Names are exact in both targets: `mdp.bronze`, etc.

# Using this project using the CLI

The Databricks workspace and IDE extensions provide a graphical interface for working
with this project. It's also possible to interact with it directly using the CLI:

1. Authenticate to your Databricks workspace, if you have not done so already:
    ```
    $ databricks configure
    ```

2. To deploy a development copy of this project, type:
    ```
    $ databricks bundle deploy --target dev
    ```
    (Note that "dev" is the default target, so the `--target` parameter
    is optional here.)

    This deploys everything that's defined for this project.

3. Similarly, to deploy a production copy, type:
   ```
   $ databricks bundle deploy --target prod
   ```

4. To run a job or pipeline, use the "run" command:
   ```
   $ databricks bundle run
   ```

5. Finally, to run tests locally, use `pytest`:
   ```
   $ uv run pytest
   ```
